# OSeduc Proposal: On-Policy OS 智能教学平台

## 1. 项目愿景

OSeduc 旨在构建一个面向本科操作系统课程的智能教学平台。平台以操作系统知识图谱为骨架，以章节式教学主线为约束，以学生的实时学习行为和实验反馈为依据，为每个学生生成可解释、可调整、可评估的个性化学习路径。

这里的 `on-policy` 指学习策略闭环：系统根据当前学生状态推荐下一步学习动作，观察学生完成情况，再更新学生画像和后续推荐策略。平台不只是课程资料网站，而是一个围绕“学什么、为什么学、怎么练、哪里错、下一步做什么”持续工作的智能导师。

## 2. 核心目标

- 为学生提供一条清晰的 OS 学习主线，同时允许根据个人薄弱点调整讲解深度、练习密度和补充路径。
- 将 OS 课程内容拆解为可计算的知识节点，使概念、章节、实验、代码能力和测试点之间形成可追踪关系。
- 将测验、实验、调试日志、对话和教师评价转化为学生画像，驱动后续学习策略。
- 为教师提供班级层面的薄弱点分析、学习进度视图和教学干预依据。
- 在使用外部 reference 仓库时严格遵守 credit、license 和学术诚信要求。

## 3. 知识图谱设计

知识图谱的基本节点是 OS 能力点和教学概念。首版建议覆盖以下主线：

- 启动与裸机执行：boot、linker、entry、SBI、QEMU。
- 执行上下文：trap、context switch、task switching。
- 内存管理：物理页、页表、Sv39、地址空间、用户缓冲区。
- 系统调用：ABI、用户态封装、内核态分发、错误处理。
- 进程与调度：task、process、fork、exec、wait、priority、stride。
- 文件系统：block device、easy-fs、inode、file descriptor、pipe。
- 同步与并发：mutex、semaphore、condition variable、thread。
- 信号与异常：signal defs、signal delivery、default action、masking。

每个知识节点至少包含：

- `id`：稳定节点标识。
- `title`：面向学生的名称。
- `chapter`：所属章节或跨章节标记。
- `type`：concept、capability、lab-skill、debug-skill、assessment-point。
- `summary`：短说明。
- `prerequisites`：先修节点。
- `learning_objectives`：学生应能完成的行为。
- `common_misconceptions`：常见误区。
- `linked_exercises`：相关测验、实验或代码任务。
- `source_refs`：参考来源、路径、commit 和 license 信息。

关系类型建议包括：

- `requires`：学习前置依赖。
- `extends`：章节能力递进。
- `implements`：实验代码实现某知识能力。
- `tests`：测验或测试覆盖某节点。
- `often_confused_with`：常混淆概念。
- `remediates`：某学习动作用于补救某薄弱点。

## 4. 学生模型与 On-Policy 闭环

平台为每个学生维护一个动态学生模型。学生模型不是单一分数，而是面向知识节点和能力维度的状态估计。

核心状态包括：

- 概念掌握度：每个知识节点的掌握概率或等级。
- 实验能力：构建、调试、阅读错误、定位内核问题、编写测试的能力。
- 最近错误：测验错题、实验失败日志、导师对话中的困惑点。
- 学习节奏：完成时间、重复尝试次数、卡住时长。
- 偏好与风险：需要更多直觉解释、更多代码例子、更多测试驱动练习等。

Policy Engine 的基本输入：

- 当前章节和课程主线。
- 知识图谱依赖关系。
- 学生模型。
- 最近学习事件和实验结果。
- 教师设置的教学目标和约束。

Policy Engine 的输出是下一步学习动作：

- 阅读一个定制讲解。
- 回看某个先修知识节点。
- 完成小测或概念辨析题。
- 修改实验代码并运行指定测试。
- 阅读一段参考实现或对照材料。
- 与智能导师进行一次针对性问答。

每次推荐都必须给出简短理由，例如：“你在 fork/wait 测试中失败，错误集中在父子关系维护，因此下一步先补 task-manage-relations，再回到 ch5 实验。”

## 5. 平台模块

### 学生端

- 章节学习地图：展示 ch1 到 ch8 的主线进度和当前所在位置。
- 知识图谱视图：展示当前章节相关节点、先修关系和薄弱节点。
- 今日学习路径：列出系统推荐的 3 到 5 个学习动作。
- 智能导师：回答问题、解释错误、生成补充讲解、提示下一步。
- 实验工作台：接入学生代码、测试结果和日志分析。
- 学习记录：展示掌握度、历史错误和已完成任务。

### 教师端

- 课程图谱管理：维护章节、知识节点、依赖关系和学习资源。
- 班级热力图：查看哪些节点普遍薄弱。
- 学生画像：查看个体进度、风险点和推荐干预。
- 作业与实验看板：汇总通过率、失败类型和常见错误。
- 策略配置：设置某章节必须完成的节点、可选深入内容和评测标准。

### 内容与评测系统

- 内容系统保存知识节点、讲解材料、测验、实验任务和来源说明。
- 评测系统统一接收 quiz、lab、log、conversation、teacher note 等事件。
- 实验反馈系统解析构建错误、测试失败、QEMU 输出和日志，映射到知识节点。
- 导师回答需要引用课程节点、实验任务或 reference 来源，避免无出处泛化。

## 6. MVP 范围

首版 MVP 不是简单资料库，而是一个可运行的完整智能导师原型。目标包括：

- 导入一版覆盖 ch1 到 ch8 的 OS 知识图谱。
- 支持学生查看章节主线、知识节点和个性化推荐。
- 支持基本测验和实验反馈事件进入学生模型。
- 根据学生表现改变下一步推荐。
- 提供智能导师问答，并能解释推荐原因。
- 提供教师端班级薄弱点视图。
- 内置 attribution/license 页面，明确 reference 仓库来源、commit、license 和复用边界。

首版可以先支持少量高价值章节闭环，例如 ch3 任务切换、ch4 地址空间、ch5 进程管理，再扩展到完整 ch1 到 ch8。

## 7. 数据与接口草案

建议首版使用 Web 前端、API 后端、Postgres 结构化存储和向量检索组件。知识图谱可先使用关系表表达，后续如果需要复杂图查询，再迁移到专门图数据库。

核心数据对象：

- `KnowledgeNode`：知识节点。
- `KnowledgeEdge`：节点关系。
- `StudentState`：学生画像快照。
- `LearningAction`：推荐学习动作。
- `PolicyDecision`：策略引擎输出及理由。
- `AssessmentEvent`：测验、实验、对话或教师评价事件。
- `SourceReference`：外部参考来源和 license 信息。

关键 API 能力：

- 获取学生当前学习路径。
- 提交测验或实验结果。
- 更新学生模型。
- 生成下一步学习动作。
- 查询知识图谱节点和依赖。
- 生成导师回答并返回引用来源。
- 教师查询班级薄弱点和学生风险列表。

## 8. Reference 与学术诚信原则

本项目使用以下本地 reference 仓库作为外部参考材料：

- `spec-driven-rust-os`
- `spec-driven-c-os`
- `fm-agent-tgrcore-reproduction`

使用原则：

- `spec-driven-rust-os` 是 GPLv3 reference。默认只做结构、规格和实验方法参考，不直接复制 GPLv3 代码进入非 GPL 产物。
- `spec-driven-c-os` 是 MIT reference。允许在保留 copyright、license notice、来源路径和 commit 的前提下复用或改写。
- `fm-agent-tgrcore-reproduction` 是混合研究材料，其中上游 `FM-Agent/` 为 Apache-2.0，仓库整体 license 需要谨慎确认。默认只做审计方法和实验报告参考。
- 所有直接复制、改写、概念借鉴和实验结论引用都必须标明来源。
- 任何报告、论文或公开发布都必须包含 reference credit 和 license 说明。

详细规则见 `REFERENCE_CREDITS.md`。

## 9. 评价指标

学习效果：

- 章节测验正确率提升。
- 实验通过率提升。
- 平均调试时间下降。
- 学生对关键概念的长期保持度。

个性化效果：

- 不同学生是否获得不同学习路径。
- 推荐是否与学生错误历史一致。
- 推荐理由是否可解释。

平台效果：

- 教师是否能发现班级共性薄弱点。
- 智能导师回答是否准确、可追溯、不越界。
- 知识图谱依赖是否合理，是否避免跳过关键先修。

工程效果：

- 学习事件和策略决策可审计。
- reference 来源可追踪。
- 平台模块边界清晰，便于逐步扩展。

## 10. 迭代路线

第一阶段：建立项目基础。

- 完成 proposal、license/credit 规则和初始 git 版本。
- 从 reference 中抽取概念层面的知识节点草案。
- 明确 ch1 到 ch8 的主线节点和依赖。

第二阶段：构建 MVP。

- 实现知识图谱导入和浏览。
- 实现学生模型和基础 policy engine。
- 实现学生端学习路径和智能导师原型。
- 实现教师端薄弱点看板。

第三阶段：接入实验闭环。

- 接入代码仓库、测试结果和日志解析。
- 将实验失败映射到知识节点。
- 让 policy engine 基于实验反馈生成补救路径。

第四阶段：评估和扩展。

- 在真实课程或模拟班级中评估学习效果。
- 扩展更多章节、测验和实验。
- 引入更精细的策略评估和教师干预工具。
