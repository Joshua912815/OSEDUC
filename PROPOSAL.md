# OSeduc Proposal: On-Policy OS 智能教学平台

## 1. 项目愿景

OSeduc 旨在构建一个面向本科操作系统课程的智能教学平台。平台以操作系统知识图谱为骨架，以章节式教学主线为约束，以学生的实时学习行为和实验反馈为依据，为每个学生生成可解释、可调整、可评估的个性化学习路径。

这里的 `on-policy` 指学习策略闭环：系统根据当前学生状态推荐下一步学习动作，观察学生完成情况，再更新学生画像和后续推荐策略。平台不只是课程资料网站，而是一个围绕“学什么、为什么学、怎么练、哪里错、下一步做什么”持续工作的智能导师。

平台可以接入 LLM API，但 LLM 不应成为无约束的“答案生成器”。LLM 在本项目中承担受控智能层：基于知识图谱、课程材料、学生状态和实验证据生成讲解、诊断、问答和反馈；学习路径决策仍由可审计的课程策略、学生模型和教师约束共同决定。

## 2. 核心目标

- 为学生提供一条清晰的 OS 学习主线，同时允许根据个人薄弱点调整讲解深度、练习密度和补充路径。
- 将 OS 课程内容拆解为可计算的知识节点，使概念、章节、实验、代码能力和测试点之间形成可追踪关系。
- 将测验、实验、调试日志、对话和教师评价转化为学生画像，驱动后续学习策略。
- 接入 LLM API，为学生提供个性化讲解、问答、实验反馈和学习路径解释，同时保证引用来源、边界控制、成本可控和教师可审计。
- 为教师提供班级层面的薄弱点分析、学习进度视图和教学干预依据。
- 在使用外部 reference 仓库时严格遵守 credit、license 和学术诚信要求。

## 3. 知识图谱设计

知识图谱的基本节点是 OS 能力点和教学概念。首版建议覆盖以下主线：

- 启动与裸机执行：boot、linker、entry、SBI、QEMU。
- 执行上下文：trap、context switch、task switching。
- 内存管理：物理页、页表、Sv39、地址空间、用户缓冲区。
- 系统调用：ABI、用户态封装、内核态分发、错误处理。
- 进程与调度：task、process、fork、exec、wait、priority、stride。
- 文件系统：block device、easy-fs、inode、file descriptor、pipe。
- 同步与并发：mutex、semaphore、condition variable、thread。
- 信号与异常：signal defs、signal delivery、default action、masking。

每个知识节点至少包含：

- `id`：稳定节点标识。
- `title`：面向学生的名称。
- `chapter`：所属章节或跨章节标记。
- `type`：concept、capability、lab-skill、debug-skill、assessment-point。
- `summary`：短说明。
- `prerequisites`：先修节点。
- `learning_objectives`：学生应能完成的行为。
- `common_misconceptions`：常见误区。
- `linked_exercises`：相关测验、实验或代码任务。
- `source_refs`：参考来源、路径、commit 和 license 信息。
- `retrieval_refs`：可供 LLM 检索和引用的课程材料片段。
- `tutor_policy`：该节点允许的提示方式、禁止直接给出的内容和推荐练习类型。

关系类型建议包括：

- `requires`：学习前置依赖。
- `extends`：章节能力递进。
- `implements`：实验代码实现某知识能力。
- `tests`：测验或测试覆盖某节点。
- `often_confused_with`：常混淆概念。
- `remediates`：某学习动作用于补救某薄弱点。

## 4. 学生模型与 On-Policy 闭环

平台为每个学生维护一个动态学生模型。学生模型不是单一分数，而是面向知识节点和能力维度的状态估计。

核心状态包括：

- 概念掌握度：每个知识节点的掌握概率或等级。
- 实验能力：构建、调试、阅读错误、定位内核问题、编写测试的能力。
- 最近错误：测验错题、实验失败日志、导师对话中的困惑点。
- 学习节奏：完成时间、重复尝试次数、卡住时长。
- 偏好与风险：需要更多直觉解释、更多代码例子、更多测试驱动练习等。

Policy Engine 的基本输入：

- 当前章节和课程主线。
- 知识图谱依赖关系。
- 学生模型。
- 最近学习事件和实验结果。
- 教师设置的教学目标和约束。

Policy Engine 的输出是下一步学习动作：

- 阅读一个定制讲解。
- 回看某个先修知识节点。
- 完成小测或概念辨析题。
- 修改实验代码并运行指定测试。
- 阅读一段参考实现或对照材料。
- 与智能导师进行一次针对性问答。

每次推荐都必须给出简短理由，例如：“你在 fork/wait 测试中失败，错误集中在父子关系维护，因此下一步先补 task-manage-relations，再回到 ch5 实验。”

## 5. LLM API 接入架构

LLM API 接入应设计为独立的 `LLM Gateway`，由平台统一调用和审计，而不是让前端或业务模块直接访问模型。这样可以集中管理 prompt、检索、权限、成本、日志和安全边界。

### LLM Gateway

- 支持多模型供应商配置，屏蔽具体 API 差异。
- 统一处理系统提示词、课程上下文、学生状态摘要和工具调用结果。
- 记录每次请求的用途、输入摘要、引用来源、输出、成本和延迟。
- 提供降级策略：模型不可用时返回课程材料链接、固定提示或教师预设反馈。

### RAG 与可追溯回答

- LLM 回答必须优先基于课程知识节点、实验说明、学生日志、测验结果和允许引用的 reference 材料。
- 每个导师回答应返回 `citations`，指向知识节点、课程材料、实验任务或 reference credit。
- 对没有可靠来源的问题，导师应明确说明不确定，并引导学生回到可验证材料或请教师确认。
- reference 仓库内容进入检索库前必须携带 license、commit 和路径元数据，避免无来源复述。

### 工具调用与实验反馈

LLM 不直接判断所有实验结果，而是调用受控工具和结构化数据：

- 日志解析器：提取 build error、test failure、QEMU 输出和 panic 信息。
- 知识映射器：把错误映射到知识节点和常见误区。
- Policy Engine：根据学生状态生成下一步动作。
- Tutor Composer：把结构化诊断转化为面向学生的解释和提示。

实验反馈应采用“提示式帮助”优先：解释错误、指出相关知识点、建议检查路径；避免直接给出完整可提交代码，除非教师明确允许。

### 教学安全与边界

- 学术诚信：导师不能帮助学生伪造学习记录、绕过作业、直接生成完整提交答案。
- 课程一致性：导师回答必须服从当前章节主线和教师设定，不鼓励跳到尚未学习的复杂替代方案。
- 个人隐私：学生代码、日志和对话进入 LLM 前应做最小化处理，避免发送无关个人信息。
- 教师可控：教师可以设置哪些节点允许代码级提示，哪些只能给概念级提示。

### 成本与观测

- 按任务类型选择模型：路径解释和简单问答用低成本模型，复杂实验诊断用强模型。
- 对重复问题、固定知识讲解和常见错误使用缓存。
- 记录 token、成本、延迟、失败率、引用覆盖率和教师/学生反馈。
- 对高风险回答进行抽样人工复核，持续改进 prompt 和检索材料。

## 6. 平台模块

### 学生端

- 章节学习地图：展示 ch1 到 ch8 的主线进度和当前所在位置。
- 知识图谱视图：展示当前章节相关节点、先修关系和薄弱节点。
- 今日学习路径：列出系统推荐的 3 到 5 个学习动作。
- 智能导师：通过 LLM API 回答问题、解释错误、生成补充讲解、提示下一步，并返回引用来源。
- 实验工作台：接入学生代码、测试结果和日志分析。
- 学习记录：展示掌握度、历史错误和已完成任务。

### 教师端

- 课程图谱管理：维护章节、知识节点、依赖关系和学习资源。
- 班级热力图：查看哪些节点普遍薄弱。
- 学生画像：查看个体进度、风险点和推荐干预。
- 作业与实验看板：汇总通过率、失败类型和常见错误。
- 策略配置：设置某章节必须完成的节点、可选深入内容和评测标准。
- LLM 规则配置：设置导师回答边界、代码提示等级、模型使用策略和需要人工复核的场景。

### 内容与评测系统

- 内容系统保存知识节点、讲解材料、测验、实验任务和来源说明。
- 评测系统统一接收 quiz、lab、log、conversation、teacher note 等事件。
- 实验反馈系统解析构建错误、测试失败、QEMU 输出和日志，映射到知识节点。
- 导师回答需要引用课程节点、实验任务或 reference 来源，避免无出处泛化。

### LLM 与策略系统

- LLM Gateway 统一接入模型 API、prompt 模板、检索上下文和工具调用。
- Retrieval Service 提供课程材料、知识节点和 reference 片段检索。
- Prompt Registry 保存不同任务的提示词版本，例如概念讲解、实验诊断、错因归纳、路径解释。
- Guardrail Service 检查回答是否越界、是否缺少引用、是否泄露完整答案。
- Conversation Memory 保存面向学习状态的摘要，不长期保存无关隐私信息。

## 7. MVP 范围

首版 MVP 不是简单资料库，而是一个可运行的完整智能导师原型。目标包括：

- 导入一版覆盖 ch1 到 ch8 的 OS 知识图谱。
- 支持学生查看章节主线、知识节点和个性化推荐。
- 支持基本测验和实验反馈事件进入学生模型。
- 根据学生表现改变下一步推荐。
- 接入至少一种 LLM API，提供基于课程材料和知识图谱的智能导师问答。
- 提供 LLM 驱动的推荐解释、概念补讲和实验错误分析，且每条回答带引用或不确定性说明。
- 提供教师端班级薄弱点视图。
- 提供教师端 LLM 边界设置，例如是否允许代码级提示、是否启用完整答案防护。
- 内置 attribution/license 页面，明确 reference 仓库来源、commit、license 和复用边界。

首版可以先支持少量高价值章节闭环，例如 ch3 任务切换、ch4 地址空间、ch5 进程管理，再扩展到完整 ch1 到 ch8。

## 8. 数据与接口草案

建议首版使用 Web 前端、API 后端、Postgres 结构化存储和向量检索组件。知识图谱可先使用关系表表达，后续如果需要复杂图查询，再迁移到专门图数据库。

核心数据对象：

- `KnowledgeNode`：知识节点。
- `KnowledgeEdge`：节点关系。
- `StudentState`：学生画像快照。
- `LearningAction`：推荐学习动作。
- `PolicyDecision`：策略引擎输出及理由。
- `AssessmentEvent`：测验、实验、对话或教师评价事件。
- `SourceReference`：外部参考来源和 license 信息。
- `LLMRequest`：模型调用用途、输入摘要、上下文来源、成本和状态。
- `TutorResponse`：导师输出、引用、置信度、边界标记和后续建议。
- `RetrievalChunk`：可被检索的课程或 reference 片段，带路径、commit、license 和节点映射。
- `GuardrailEvent`：越界、缺引用、疑似直接答案、隐私风险等检查结果。

关键 API 能力：

- 获取学生当前学习路径。
- 提交测验或实验结果。
- 更新学生模型。
- 生成下一步学习动作。
- 查询知识图谱节点和依赖。
- 生成导师回答并返回引用来源。
- 教师查询班级薄弱点和学生风险列表。
- 发起 LLM 导师请求，并返回结构化回答、引用和安全标记。
- 查询模型调用成本、失败率和引用覆盖率。
- 配置课程级 LLM 使用策略和提示边界。

## 9. Reference 与学术诚信原则

本项目使用以下本地 reference 仓库作为外部参考材料：

- `spec-driven-rust-os`
- `spec-driven-c-os`
- `fm-agent-tgrcore-reproduction`

使用原则：

- `spec-driven-rust-os` 是 GPLv3 reference。默认只做结构、规格和实验方法参考，不直接复制 GPLv3 代码进入非 GPL 产物。
- `spec-driven-c-os` 是 MIT reference。允许在保留 copyright、license notice、来源路径和 commit 的前提下复用或改写。
- `fm-agent-tgrcore-reproduction` 是混合研究材料，其中上游 `FM-Agent/` 为 Apache-2.0，仓库整体 license 需要谨慎确认。默认只做审计方法和实验报告参考。
- 所有直接复制、改写、概念借鉴和实验结论引用都必须标明来源。
- 任何报告、论文或公开发布都必须包含 reference credit 和 license 说明。
- LLM 生成内容不能掩盖来源：如果回答、讲解或实验反馈受 reference 材料影响，必须保留 source metadata 并在可见层面给出引用。
- LLM 不得把 GPLv3 reference 的实现细节改写后伪装为原创非 GPL 代码。

详细规则见 `REFERENCE_CREDITS.md`。

## 10. 评价指标

学习效果：

- 章节测验正确率提升。
- 实验通过率提升。
- 平均调试时间下降。
- 学生对关键概念的长期保持度。

个性化效果：

- 不同学生是否获得不同学习路径。
- 推荐是否与学生错误历史一致。
- 推荐理由是否可解释。

平台效果：

- 教师是否能发现班级共性薄弱点。
- 智能导师回答是否准确、可追溯、不越界。
- 知识图谱依赖是否合理，是否避免跳过关键先修。

LLM 效果：

- 导师回答是否引用充分。
- 实验反馈是否能定位到正确知识节点。
- 是否减少学生卡住时间而不直接替代学生完成作业。
- 模型成本、延迟和失败率是否可控。
- 教师抽样复核的高风险回答通过率。

工程效果：

- 学习事件和策略决策可审计。
- reference 来源可追踪。
- LLM 调用、prompt 版本、检索材料和回答边界可审计。
- 平台模块边界清晰，便于逐步扩展。

## 11. 迭代路线

第一阶段：建立项目基础。

- 完成 proposal、license/credit 规则和初始 git 版本。
- 从 reference 中抽取概念层面的知识节点草案。
- 明确 ch1 到 ch8 的主线节点和依赖。
- 设计 LLM Gateway、RAG 元数据和导师回答边界。

第二阶段：构建 MVP。

- 实现知识图谱导入和浏览。
- 实现学生模型和基础 policy engine。
- 实现学生端学习路径和智能导师原型。
- 实现教师端薄弱点看板。
- 接入首个 LLM API，支持概念问答、路径解释和引用返回。

第三阶段：接入实验闭环。

- 接入代码仓库、测试结果和日志解析。
- 将实验失败映射到知识节点。
- 让 policy engine 基于实验反馈生成补救路径。
- 让 LLM 基于结构化实验诊断生成提示式反馈，避免直接给完整答案。

第四阶段：评估和扩展。

- 在真实课程或模拟班级中评估学习效果。
- 扩展更多章节、测验和实验。
- 引入更精细的策略评估和教师干预工具。
- 建立 LLM 回答质量、成本和安全边界的持续评估流程。
